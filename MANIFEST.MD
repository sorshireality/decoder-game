## Геймплей (текущая реализация)

- Жанр: угадывание секретной последовательности цветов по позициям.
- Проверка попытки: считается только количество цветов, совпавших **по правильной позиции** (`correct / total`).
- По умолчанию секрет собирается из уникальных цветов (через `shuffle` + `slice`).
- Исключение: в `Stage 3 (Repeats)` повторы цветов в секретном коде **разрешены**.
- Ввод: выбрать цвет в палитре -> нажать слот.
- Подтверждение доступно только когда все слоты заполнены.
- История попыток отображает:
  - набор выбранных цветов
  - результат в формате `X/N`

### Daily Challenge (MVP)

- Один общий daily-челлендж в день для всех игроков (детерминированный seed по дате, `UTC`, `daily:<YYYY-MM-DD>:v1`).
- Одна **зачетная** попытка в день (win или loss закрывает daily до следующего дня).
- Daily запускается с `home-screen` кнопкой `Daily / Щоденна`.
- Daily сейчас использует расширяемый пул шаблонов `DAILY_CHALLENGE_POOL` (`kind: preset_level`) и выбирает из готовых уровней/механик (`Stage 1/2/3`, включая `+1` и `Repeats`).
- Формула награды за прохождение daily:
  - `15` базовых очков
  - `+5` за каждую неиспользованную попытку (`attemptsLeft`)
- Daily результат и факт использования зачетной попытки хранятся локально в `decoder_daily_state` (MVP).

## Уровни и прогрессия

- Всего уровней: **41**
- Стадий: **6**
- В коде добавлен `LEVELS v2` каркас (backward-compatible):
  - `LEVEL_PRESETS_V1` -> исходный список текущих уровней
  - `LEVELS_V2` -> расширенный формат уровня (`feature`, `featureConfig`, `scoring`, `mode`, `stageLevel`)
  - `LEVELS` пока остается legacy-совместимым flattened-view (`lvl.game`) для сохранения текущего поведения

### Stage 1 (уровни 1-10)

- Без лишнего цвета в палитре (`paletteColors === colors`)

### Stage 2 (уровни 11-15)

- В палитре есть **1 лишний цвет** (`paletteColors = colors + 1`)

### Stage 3 (уровни 16-23)

- Механика `Repeats`: в секретном коде разрешены повторяющиеся цвета (`feature = repeats`, `featureConfig.allowDuplicates = true`)
- Палитра без лишнего цвета (на текущем этапе)

### Stage 4 (уровни 24-29)

- Механика `Fog`: в истории попыток видны только последние `3` попытки (`feature = fog`, `featureConfig.visibleHistory = 3`)

### Stage 5 (уровни 30-35)

- Механика `Freeze`: начиная со 2-го хода после каждого подтверждения случайная комірка замораживается на следующий ход (`feature = freeze`)
- Замороженную комірку нельзя менять; ability не может открыть замороженную комірку

### Stage 6 (уровни 36-41)

- Механика `Chain`: один уровень состоит из серии мини-раундов (`feature = chain`)
- Каждый следующий мини-раунд:
  - имеет на 1 попытку меньше
  - стартует с дополнительной заранее открытой позицией
- Проигрыш любого мини-раунда = проигрыш всего уровня

### LEVELS v2 (подготовка к новым стадиям)

`LEVELS_V2` уже используется как основной источник конфигурации уровней (с legacy flattened `LEVELS` для совместимости UI/логики). В нем:

- `feature` (`none`, `repeats`, `fog`, `freeze`, `chain`)
- `featureConfig`
- `mode` (`classic` / `plus_one`)
- `scoring` (`stageBasePoints`, `levelCoefficient`)
- `stageLevel`, `globalLevel`

Stage 1/2 поведение сохранено. Stage 3/4/5/6 уже используют `LEVELS_V2.feature` для механик `repeats/fog/freeze/chain`.
`Double Code` и следующие стадии/миксы пока не реализованы и должны добавляться через тот же механизм `feature + featureConfig`.

### Разблокировка

- Прогресс открытых уровней хранится как `unlockedLevel`.
- При победе на текущем максимальном уровне открывается следующий.
- После прохождения уровня 10 показывается уведомление об открытии Stage 2.

## Способность (ability)

- Разблокируется после прохождения уровня **6** (`ABILITY_UNLOCK_LEVEL = 6`).
- Кнопка: `Reveal a position / Відкрити позицію`.
- Использование:
  - открывает случайную неоткрытую позицию
  - автоматически подставляет правильный цвет в текущую попытку
  - доступна **1 раз за уровень** (сбрасывается при старте уровня)
- Открытая позиция сохраняется между попытками в рамках уровня.

PvP UX (текущее состояние):

  - свой прогресс (`Host`/`Guest`)
  - прогресс соперника
  - индикаторы `wins needed`:

    - оранжевый = текущая цель
    - зеленый = уже выигранный раунд
- После выигрыша/проигрыша локального раунда не показывается обычный solo result screen:
  - используется быстрый `pvp-round-screen` с `Next` / `Retry`
- Финал матча показывает `Victory/Defeat` через `result-screen` с кнопкой возврата в PvP.
- PvP-лобби упрощено:
  - главный экран PvP с 2 кнопками: `Поиск матча` / `Создать`
  - отдельные вложенные подэкраны (в рамках `pvp-screen`) для create/join flow
- На главном экране PvP скрыты `Google login/logout` и `Change Nick` (они не участвуют в основном дуэльном UX; Google-вход остается на `home-screen`, ник задается модалкой при входе в PvP при отсутствии).
- При первом входе в PvP (если нет сохраненного ника) показывается модальное окно с предложением добавить никнейм.
- Никнейм пока хранится локально в `localStorage` (без записи в БД) и используется в PvP UI для отображения локального игрока.
- Блок `Connection` убран из контента PvP-экрана; вместо него статус подключения показывается компактным цветным chip в правом верхнем углу header.
- Проверка PvP backend-конфига кешируется на клиенте (короткий TTL), чтобы статус chip не мигал во время циклического автопоиска.
- В `Join` добавлен `Matchmaking`:
  - `Any room`
  - `By settings` (mode/colors/best_of)
  - авто-подключение к первой подходящей комнате в статусе `waiting_for_opponent`
- Добавлен `автопоиск матча` (циклический поиск с кнопкой остановки) в `Join`-экране.
- Если подходящих комнат не найдено и игрок ищет матч через `Matchmaking`, клиент может автоматически создать комнату (MVP-эвристика) и перевести host в экран ожидания.
- Matchmaking дополнительно фильтрует слишком старые комнаты (TTL по `created_at`) чтобы снизить риск подключения к “зависшим” комнатам после аварийного закрытия клиента.
- Для автопоиска добавлен отдельный экран ожидания внутри `pvp-screen`:
  - анимированный индикатор поиска
  - `Cancel`
  - `Join by Code`
- Для формата `best of N` показывается подсказка в стиле `до X побед` (majority of rounds).
- Добавлена защита от случайного `tap-through` на экранах результатов (короткий input lock + задержка кнопки выхода в PvP финале).
- После создания комнаты host получает отдельный экран ожидания:
  - статус `Room Created`
  - показ `room code`
  - ожидание подключения соперника
  - кнопка `Cancel`
- Matchmaking join усилен атомарным захватом комнаты (`status=waiting_for_opponent` + `guest_player_id is null`) для снижения гонок при автопоиске.
- Для join flow добавлен fallback re-read комнаты после `UPDATE ... RETURNING`, чтобы снизить риск ложного `Room unavailable` при особенностях RLS/возврата строки.
- При локальном выходе из PvP UI принудительно возвращается в `PVP` меню (чтобы не оставался пустой экран после закрытия комнаты/матча).

## Auth / Profile / Score (MVP)

- Доступны режимы `guest + Google Auth` (через Supabase Auth).
- Кнопка входа Google вынесена на `home-screen` рядом с переключателем языка.
- После логина выполняется merge локального прогресса (`decoder_progress`) в облачный `player_progress`.
- Cloud sync прогресса/очков и toast `Progress synced` выполняются только на значимых auth-событиях (`INITIAL_SESSION/SIGNED_IN/USER_UPDATED`), без постоянного визуального спама на `TOKEN_REFRESHED`.
- Локальный никнейм используется как fallback и для PvP UI; при create/join в PvP пишется в `room_players.nickname` (после применения SQL миграции).
- Main Game очки (MVP):
  - начисляются за победу в solo-уровне
  - начисляются **только за первое прохождение конкретного уровня** (`first clear`)
  - перепрохождение уровня не добавляет рейтинг/очки повторно
  - формула: `stage base points * level coefficient`
  - отображаются на `home-screen`
  - синхронизируются в `player_scores.main_game_score` и `player_scores.rating_score` при наличии Google-аккаунта
  - защита от повторной выдачи для Main Game сейчас хранится локально в `decoder_progress.awardedMainLevels` (MVP)
- Глобальный рейтинг (MVP):
  - отдельный экран `leaderboard-screen`
  - `Top 10` по `player_scores.rating_score`
  - загрузка `player_scores` + `profiles` (nickname/email)
  - подсветка текущего авторизованного игрока
- Для работы глобального рейтинга между разными пользователями нужны публичные `SELECT`-policy на `profiles` и `player_scores` (добавлены в `SUPABASE_AUTH_MVP.sql`, требуется выполнить SQL повторно).
- Daily очки (MVP):
  - при успешном прохождении начисляются в `player_scores.daily_score` и `rating_score` (если игрок авторизован через Google)
  - локально также увеличивают отображаемый счет на `home-screen`

## Известные архитектурные ограничения (текущие)

- Нет бэкенда и синхронизации между устройствами.
- Нет полноценного PvP-геймплея и сетевой синхронизации раундов.
- Нет серверной валидации (вся логика на клиенте).
- Прогресс привязан к браузеру/устройству через `localStorage`.

Актуализация после PvP MVP шага:

- Есть создание и подключение к комнате по `room code` через Supabase.
- Добавлено наблюдение за комнатой после `create/join`: `Supabase Realtime` (если доступен) + polling fallback.
- После перезагрузки страницы PvP-сессия комнаты восстанавливается из `localStorage` и подтягивается из БД.
- Есть базовый PvP match-loop на скорость:
  - прогресс по победам (`best of`)
  - pre-match `VS` экран перед стартом матча (показывает локального игрока и соперника, плюс рейтинг если доступен)
  - быстрый переход `Next/Retry`
  - финальный `Victory/Defeat`
- Пока нет синхронизации попыток/истории соперника внутри игрового поля (только прогресс/счет).
- PvP очки (MVP) начисляются в `player_scores.pvp_score` и общий `rating_score` при финале матча:
  - победа: `+20`
  - поражение: `+5`
  - начисление защищено от дублей на повторных realtime-обновлениях/повторном показе результата
- Для пользователя Supabase-конфиг должен приходить через `Vercel /api/config`.
- На PvP-экране оставлен скрытый `Dev Config (fallback)` для локальной отладки.

## PvP MVP Design (зафиксированный план)

Этот раздел фиксирует целевую MVP-концепцию PvP. До реализации считается планом, после реализации должен быть обновлен до фактического состояния.

### Целевая платформа для MVP

- Frontend/hosting: `Vercel`
- Backend/data/realtime: `Supabase`
- Цель: сохранить приложение легковесным, без тяжелого серверного слоя
- Публичный `Supabase URL + anon key` отдается через `Vercel /api/config`, а не вводится пользователем вручную.

### Роли игрока в PvP

- `Игрок 1`: выбирает `PVP -> ведущий` (создает комнату)
- `Игрок 2`: выбирает `PVP -> подключиться` (подключается к комнате)

### Важно: поиск комнат по локальной сети (LAN)

- Для веб-приложения в браузере **не рассчитывать на LAN discovery** как основной сценарий.
- В MVP использовать один из легковесных вариантов:
  - список открытых комнат (`lobby`)
  - подключение по `room code`
  - подключение по `invite link`

Если когда-либо добавляется иной способ поиска/подключения, это обязательно отражается в `MANIFEST.MD`.

### Настройки матча (задает ведущий)

Ведущий перед созданием комнаты выбирает:

- количество цветов (`colors`)
- режим:
  - `classic`
  - `+1 decoy` (аналог Stage 2 основной игры: в палитре 1 лишний цвет)
- формат победы:
  - `best of 1`
  - `best of 3`
  - `best of 5`

### Поведение комнаты (MVP)

- После выбора настроек создается комната со статусом ожидания второго игрока.
- После подключения второго игрока матч переходит в состояние готовности/старта.
- Игровой интерфейс каждого игрока остается максимально близким к текущему single-player UI.

Текущее фактическое покрытие:

- Реализовано: создание комнаты (host) с сохранением настроек в БД.
- Реализовано: подключение к комнате по `room code` (guest), перевод комнаты в `ready`.
- Реализовано: создание `room_players` и `match_state` (MVP).
- Реализовано: автослежение за статусом комнаты (`Realtime + polling fallback`) и восстановление комнаты после reload.
- Реализовано: автостарт локального раунда на обоих клиентах при статусе комнаты `ready` (через сопоставление с существующими preset-levels).
- Реализовано: PvP competition UI в `game-screen` (свой/чужой прогресс по `wins needed`).
- Реализовано: быстрый PvP round flow (`Next/Retry`) без solo result screen между раундами.
- Реализовано: финальный PvP `Victory/Defeat` экран.
- Реализовано: явный `Leave/Close Room` flow (из лобби и из игрового экрана), очистка локальной PvP-сессии.
- Реализовано: базовый перевод PvP UI (`EN`/`UK`) для ключевых экранов/кнопок/форм.
- Реализовано: `guest + Google Auth` UI в PvP-лобби (MVP).
- Реализовано: синхронизация локального прогресса в облачный `player_progress` при логине (merge local->cloud).
- Реализовано: подготовка PvP никнеймов соперника через `room_players.nickname` (после применения SQL-миграции).
- Реализовано: `Rematch` в той же комнате после PvP матча (запрос / принять / отклонить) через поля `match_state.rematch_status/rematch_requested_by`.
- Не реализовано пока: синхронизация попыток/истории соперника в реальном времени внутри игрового поля и пользовательские ники в БД/UI.

### PvP UI (MVP)

- В игровом экране появляется верхняя плашка прогресса соперника.
- Пример отображения: `3/5` (угадано соперником при текущей длине кода 5).
- Основной UI ввода своей попытки остается таким же, как в обычной игре.

### Рекомендуемые состояния матча (MVP)

- `waiting_for_opponent`
- `ready`
- `in_round`
- `round_finished`
- `match_finished`
- `abandoned` (если один игрок вышел/отвалился)

### Рекомендуемый минимум данных для PvP (MVP)

- `rooms`
  - код комнаты
  - статус
  - настройки матча (`colors`, `mode`, `best_of`)
  - кто ведущий
- `room_players`
  - игрок
  - комната
  - статус подключения/готовности
- `match_state` / `round_state`
  - текущий раунд
  - счет по раундам
  - прогресс игроков в текущем раунде (`correct/total`)

### Правила качества для PvP-реализации

- Не ломать существующий single-player режим.
- Переиспользовать текущую игровую механику и UI, где возможно.
- Сохранять минимальный вес фронта и простую архитектуру.
- Все изменения PvP-потока, экранов, сетевых состояний и схем данных отражать в `MANIFEST.MD`.
