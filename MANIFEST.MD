## Текущий статус проекта (фактическое состояние)

- Тип проекта: статическая браузерная игра / PWA.
- Стек: `HTML + CSS + Vanilla JavaScript` (без фреймворков и без сборщика).
- Backend: отсутствует.
- Backend (игровой): отсутствует, но добавлена клиентская интеграция с `Supabase` для PvP MVP-лобби и `Vercel` serverless endpoint для публичного конфига.

## Геймплей (текущая реализация)

- Жанр: угадывание секретной последовательности цветов по позициям.
- Проверка попытки: считается только количество цветов, совпавших **по правильной позиции** (`correct / total`).
- Повторы цветов в коде не используются: секрет собирается из уникальных цветов (через `shuffle` + `slice`).
- Ввод: выбрать цвет в палитре -> нажать слот.
- Подтверждение доступно только когда все слоты заполнены.
- История попыток отображает:
  - набор выбранных цветов
  - результат в формате `X/N`

## Уровни и прогрессия

- Всего уровней: **15**
- Стадий: **2**

### Stage 1 (уровни 1-10)

- Без лишнего цвета в палитре (`paletteColors === colors`)

### Stage 2 (уровни 11-15)

- В палитре есть **1 лишний цвет** (`paletteColors = colors + 1`)

### Разблокировка

- Прогресс открытых уровней хранится как `unlockedLevel`.
- При победе на текущем максимальном уровне открывается следующий.
- После прохождения уровня 10 показывается уведомление об открытии Stage 2.

## Способность (ability)

- Разблокируется после прохождения уровня **6** (`ABILITY_UNLOCK_LEVEL = 6`).
- Кнопка: `Reveal a position / Відкрити позицію`.
- Использование:
  - открывает случайную неоткрытую позицию
  - автоматически подставляет правильный цвет в текущую попытку
  - доступна **1 раз за уровень** (сбрасывается при старте уровня)
- Открытая позиция сохраняется между попытками в рамках уровня.

- Переходы между экранами через `showScreen(id)`.
- Кнопка назад:
  - из выбора уровней -> на главный экран
  - из игры -> на экран выбора уровней
- Результат:
  - `Next Level` / `Play Again` / `Try Again`
  - `All Levels`

PvP UX (текущее состояние):

- В `game-screen` отображается компактный PvP-блок прогресса (правый верхний угол):
  - свой прогресс (`Host`/`Guest`)
  - прогресс соперника
  - индикаторы `wins needed`:
    - серый = еще не достигнут
    - оранжевый = текущая цель
    - зеленый = уже выигранный раунд
- После выигрыша/проигрыша локального раунда не показывается обычный solo result screen:
  - используется быстрый `pvp-round-screen` с `Next` / `Retry`
- Финал матча показывает `Victory/Defeat` через `result-screen` с кнопкой возврата в PvP.
- PvP-лобби упрощено:
  - главный экран PvP с 2 кнопками: `Создать` / `Подключиться`
  - отдельные вложенные подэкраны (в рамках `pvp-screen`) для create/join flow
- При первом входе в PvP (если нет сохраненного ника) показывается модальное окно с предложением добавить никнейм.
- Никнейм пока хранится локально в `localStorage` (без записи в БД) и используется в PvP UI для отображения локального игрока.
- Блок `Connection` убран из контента PvP-экрана; вместо него статус подключения показывается компактным цветным chip в правом верхнем углу header.
- Для формата `best of N` показывается подсказка в стиле `до X побед` (majority of rounds).
- Добавлена защита от случайного `tap-through` на экранах результатов (короткий input lock + задержка кнопки выхода в PvP финале).

## PWA / Производительность / Легковесность

- Без внешних библиотек и тяжелых зависимостей.
- Все ассеты статические.
- `sw.js` использует cache-first стратегию для основных файлов.
- `sw.js` использует `network-first` для app shell (`index.html`, `app.js`, `style.css`, `manifest.json`) и `cache-first` для прочих статических ассетов, чтобы уменьшить риск устаревшей версии после деплоя.
- UI ориентирован на мобильные устройства (`100dvh`, safe-area, touch-friendly controls).
- Это соответствует целевому требованию: **максимально легковесно и шустро**.
- Дополнительно внесены мобильные оптимизации отклика:
  - `touch-action: manipulation` для интерактивных элементов
  - убрана анимация появления строк истории (снижает нагрузку на mobile Safari)
  - ослаблены тени на touch-устройствах

## Известные архитектурные ограничения (текущие)

- Нет бэкенда и синхронизации между устройствами.
- Нет полноценного PvP-геймплея и сетевой синхронизации раундов.
- Нет серверной валидации (вся логика на клиенте).
- Прогресс привязан к браузеру/устройству через `localStorage`.

Актуализация после PvP MVP шага:

- Есть создание и подключение к комнате по `room code` через Supabase.
- Добавлено наблюдение за комнатой после `create/join`: `Supabase Realtime` (если доступен) + polling fallback.
- После перезагрузки страницы PvP-сессия комнаты восстанавливается из `localStorage` и подтягивается из БД.
- Есть базовый PvP match-loop на скорость:
  - прогресс по победам (`best of`)
  - быстрый переход `Next/Retry`
  - финальный `Victory/Defeat`
- Пока нет синхронизации попыток/истории соперника внутри игрового поля (только прогресс/счет).
- Для пользователя Supabase-конфиг должен приходить через `Vercel /api/config`.
- На PvP-экране оставлен скрытый `Dev Config (fallback)` для локальной отладки.

## PvP MVP Design (зафиксированный план)

Этот раздел фиксирует целевую MVP-концепцию PvP. До реализации считается планом, после реализации должен быть обновлен до фактического состояния.

### Целевая платформа для MVP

- Frontend/hosting: `Vercel`
- Backend/data/realtime: `Supabase`
- Цель: сохранить приложение легковесным, без тяжелого серверного слоя
- Публичный `Supabase URL + anon key` отдается через `Vercel /api/config`, а не вводится пользователем вручную.

### Роли игрока в PvP

- `Игрок 1`: выбирает `PVP -> ведущий` (создает комнату)
- `Игрок 2`: выбирает `PVP -> подключиться` (подключается к комнате)

### Важно: поиск комнат по локальной сети (LAN)

- Для веб-приложения в браузере **не рассчитывать на LAN discovery** как основной сценарий.
- В MVP использовать один из легковесных вариантов:
  - список открытых комнат (`lobby`)
  - подключение по `room code`
  - подключение по `invite link`

Если когда-либо добавляется иной способ поиска/подключения, это обязательно отражается в `MANIFEST.MD`.

### Настройки матча (задает ведущий)

Ведущий перед созданием комнаты выбирает:

- количество цветов (`colors`)
- режим:
  - `classic`
  - `+1 decoy` (аналог Stage 2 основной игры: в палитре 1 лишний цвет)
- формат победы:
  - `best of 1`
  - `best of 3`
  - `best of 5`

### Поведение комнаты (MVP)

- После выбора настроек создается комната со статусом ожидания второго игрока.
- После подключения второго игрока матч переходит в состояние готовности/старта.
- Игровой интерфейс каждого игрока остается максимально близким к текущему single-player UI.

Текущее фактическое покрытие:

- Реализовано: создание комнаты (host) с сохранением настроек в БД.
- Реализовано: подключение к комнате по `room code` (guest), перевод комнаты в `ready`.
- Реализовано: создание `room_players` и `match_state` (MVP).
- Реализовано: автослежение за статусом комнаты (`Realtime + polling fallback`) и восстановление комнаты после reload.
- Реализовано: автостарт локального раунда на обоих клиентах при статусе комнаты `ready` (через сопоставление с существующими preset-levels).
- Реализовано: PvP competition UI в `game-screen` (свой/чужой прогресс по `wins needed`).
- Реализовано: быстрый PvP round flow (`Next/Retry`) без solo result screen между раундами.
- Реализовано: финальный PvP `Victory/Defeat` экран.
- Реализовано: явный `Leave/Close Room` flow (из лобби и из игрового экрана), очистка локальной PvP-сессии.
- Реализовано: базовый перевод PvP UI (`EN`/`UK`) для ключевых экранов/кнопок/форм.
- Не реализовано пока: синхронизация попыток/истории соперника в реальном времени внутри игрового поля и пользовательские ники в БД/UI.

### PvP UI (MVP)

- В игровом экране появляется верхняя плашка прогресса соперника.
- Пример отображения: `3/5` (угадано соперником при текущей длине кода 5).
- Основной UI ввода своей попытки остается таким же, как в обычной игре.

### Рекомендуемые состояния матча (MVP)

- `waiting_for_opponent`
- `ready`
- `in_round`
- `round_finished`
- `match_finished`
- `abandoned` (если один игрок вышел/отвалился)

### Рекомендуемый минимум данных для PvP (MVP)

- `rooms`
  - код комнаты
  - статус
  - настройки матча (`colors`, `mode`, `best_of`)
  - кто ведущий
- `room_players`
  - игрок
  - комната
  - статус подключения/готовности
- `match_state` / `round_state`
  - текущий раунд
  - счет по раундам
  - прогресс игроков в текущем раунде (`correct/total`)

### Правила качества для PvP-реализации

- Не ломать существующий single-player режим.
- Переиспользовать текущую игровую механику и UI, где возможно.
- Сохранять минимальный вес фронта и простую архитектуру.
- Все изменения PvP-потока, экранов, сетевых состояний и схем данных отражать в `MANIFEST.MD`.
